# MCP Central — Development Guidelines

This document defines the authoritative development guidelines for MCP Central. All contributors
and AI agents working on this codebase must follow these rules. Consistency, safety, and
debuggability are the primary goals.

---

## Current Implementation Snapshot

MCP Central currently has a working FastAPI backend, React/Vite frontend, Docker image, SQLite
persistence, MCP stdio process manager, ZIP deployments, a single-file server editor, API keys,
groups, logs, and unified MCP proxy endpoints.

Active implemented user flows:
- Admin login with JWT authentication.
- Dashboard, Servers, Groups, API Keys, Logs, Upload/Deploy, and Endpoints pages.
- ZIP MCP server upload through `POST /api/v1/upload`.
- Single-file Python MCP server creation through `POST /api/v1/upload/single-file`.
- Per-server lifecycle actions: start, stop, restart.
- Per-server runtime parameter editing through `env_vars`; values are injected into the subprocess
  environment and require a restart to affect a running server.
- Light/dark theme toggle using Tailwind v4 class-based dark mode via `frontend/src/index.css`.

Operational notes for future sessions:
- The container service is named `hub`; the container is named `mcp-central`.
- After every code or configuration change, rebuild the container before considering the task done.
- Uvicorn must listen on container port `8000`; host exposure is controlled by `HUB_PORT` in
  `.env` through the compose mapping `${HUB_PORT:-8000}:8000`.
- In the current local environment, the app has been observed at `http://localhost:9000` because
  `HUB_PORT=9000` is set locally. Do not assume `8080`.
- Docker health checks use the internal URL `http://localhost:8000/api/health` from inside the
  container. Host checks should use `http://localhost:<HUB_PORT>/api/health`.
- The compose environment sets `TZ=${TZ:-Europe/Madrid}`. Container-local commands such as `date`
  should show Madrid time; API timestamps may still be emitted in UTC where code explicitly uses
  timezone-aware UTC datetimes.
- `frontend/dist/` is generated by `pnpm build`; avoid running backend tests concurrently with a
  frontend build because `create_app()` mounts `frontend/dist/assets` when it exists.
- `servers/` and `data/` are runtime volumes and must not be committed.

## 1. Project Structure

```
mcp-central/
├── hub/                  # Python FastAPI backend
│   ├── api/              # Route handlers (one file per resource domain)
│   ├── mcp/              # MCP protocol layer (proxy, router, namespace)
│   ├── process/          # MCP server lifecycle (manager, sandbox, health)
│   ├── auth/             # Authentication (JWT admin, API keys)
│   ├── models/           # SQLAlchemy ORM models + Pydantic schemas
│   └── migrations/       # Alembic migration files
├── frontend/             # React + Vite + shadcn/ui
│   └── src/
│       ├── pages/        # One file per UI page
│       ├── components/   # Reusable UI components
│       ├── hooks/        # Custom React hooks
│       ├── store/        # Zustand state stores
│       └── lib/          # Utilities, API client
├── tests/
│   ├── unit/             # Pure unit tests (no I/O)
│   ├── integration/      # Tests that use the DB or start processes
│   └── e2e/              # End-to-end API tests (httpx + pytest)
├── servers/              # Deployed MCP server packages (runtime, git-ignored contents)
├── data/                 # Persistent volume (SQLite DB + log files)
├── docs/                 # User-facing documentation
├── docker-compose.yml
├── Dockerfile
├── entrypoint.sh
└── AGENTS.md             # This file
```

**Rules:**
- Never put business logic in route handlers. Routes validate input and delegate to service/manager classes.
- Never import from `api/` into `process/` or `mcp/`. Dependency direction: `api → process/mcp/auth/models`.
- Every new module must have a corresponding `__init__.py` that exports its public interface.

## 2. Language & Runtime

### Python (hub backend)
- **Version:** Python 3.12 minimum.
- **Package manager:** `uv` for the hub; per-server venvs created via `venv` stdlib module.
- **Formatter:** `ruff format` (line length 100).
- **Linter:** `ruff check` with rules: `E, F, I, N, UP, B, A, C4, PT, RUF`.
- **Type checker:** `mypy --strict` on all hub source files.
- **All functions must have type annotations.** No `Any` unless explicitly justified with a comment.
- **Async-first:** All I/O (DB, subprocesses, file reads) must use `async`/`await`. No blocking calls on the event loop.

### TypeScript / React (frontend)
- **Version:** Node.js 20 LTS, TypeScript 5.x.
- **Package manager:** `pnpm`.
- **Formatter/Linter:** ESLint + Prettier (configured in repo).
- **Component style:** Functional components only. No class components.
- **State:** Zustand for global state, `useState`/`useReducer` for local component state.
- **No `any` in TypeScript.** Use `unknown` and narrow types explicitly.

## 3. Testing — Progressive Feature Validation

> **This is non-negotiable.** Every feature added must have tests written alongside it,
> not after. The test suite must pass before moving to the next phase.

### Rules
1. **Every new API endpoint** gets at least one happy-path and one error-path test in `tests/e2e/`.
2. **Every new service/manager class** gets unit tests in `tests/unit/`.
3. **Every new SQLAlchemy model** gets a migration + a round-trip test (create, read, update, delete).
4. **Process manager** tests use a mock MCP server subprocess (a simple echo script) — never spawn real servers in unit tests.
5. **Run the full test suite** after completing each sub-task before marking it done.
6. **Minimum coverage target:** 80% for `hub/` code. Use `pytest-cov` to verify.

### Test Stack
- `pytest` + `pytest-asyncio` for async tests.
- `httpx` + `pytest` for E2E API tests (using FastAPI's `ASGITransport`).
- `pytest-cov` for coverage reports.
- `factory-boy` for test data factories.

### Running Tests
```bash
# From repo root
uv run pytest tests/ -v --cov=hub --cov-report=term-missing

# Single phase
uv run pytest tests/unit/ -v
uv run pytest tests/e2e/ -v
```

### Progressive Checkpoints
Each phase ends with a checkpoint command that must succeed before the next phase begins:

| Phase | Checkpoint |
|-------|-----------|
| 1 | `uv run pytest tests/unit/test_models.py tests/e2e/test_auth.py -v` |
| 2 | `uv run pytest tests/unit/test_process_manager.py tests/e2e/test_servers_api.py -v` |
| 3 | `uv run pytest tests/e2e/test_mcp_proxy.py -v` |
| 4 | `uv run pytest tests/e2e/test_api_keys.py tests/unit/test_sandbox.py -v` |
| 5 | `uv run pytest tests/e2e/test_upload.py -v` |
| 6 | Frontend from `frontend/`: `pnpm test` and `pnpm build` |
| 7 | `docker compose build && docker compose up -d`, then `curl -f http://localhost:<HUB_PORT>/api/health` |

Do not run backend tests at the same time as `pnpm build`; the Vite build temporarily replaces
`frontend/dist/assets`, which can make FastAPI static mounting fail during test app startup.

## 4. Error Handling & Observability

### Core Principle: Never Swallow Errors

- **No bare `except Exception: pass`** — ever. If you catch an exception, you must either re-raise it,
  log it with full context, or convert it to a structured error response.
- **No generic error messages** like `"Something went wrong"`. Always include: what failed, which
  component failed, and the original exception message.

### Stack Traces
- MCP server subprocess stderr is captured line-by-line and stored with the server name + timestamp.
- Tracebacks are **never truncated** in API responses, log files, or the UI.
- Every JSON error response from the hub must include a `traceback` field when the error originated
  from an MCP server process. Format:

```json
{
  "error": {
    "code": -32603,
    "message": "Server 'my-server' raised an unhandled exception",
    "data": {
      "server": "my-server",
      "traceback": "<full traceback string>",
      "exit_code": 1,
      "timestamp": "2026-05-04T12:00:00Z"
    }
  }
}
```

- MCP server processes are launched with a wrapper that installs `sys.excepthook` to guarantee
  stderr output even when the server's own code swallows exceptions.
- `warnings.simplefilter("always")` is set in the wrapper to prevent suppression.

### Logging
- Use Python's `structlog` library for structured JSON logging in the hub.
- Every log entry must include: `timestamp`, `level`, `logger` (module name), `event`, and any
  relevant context fields (e.g., `server_name`, `request_id`).
- Log levels:
  - `DEBUG`: detailed internal state (disabled in production by default).
  - `INFO`: normal lifecycle events (server started, request received).
  - `WARNING`: recoverable issues (server restart attempt, config mismatch).
  - `ERROR`: failures that affect functionality (server crashed, DB write failed).
  - `CRITICAL`: hub-level failures requiring immediate attention.
- Never log secrets (API keys, passwords, tokens). Use `[REDACTED]` placeholder.

## 5. Security

- **No hardcoded secrets.** All secrets come from environment variables.
- **API keys are stored as SHA-256 hashes** in the database. The plaintext is shown exactly once
  at creation time and never stored.
- **Per-server `env_vars` are currently stored in cleartext** in SQLite. This is a known gap
  tracked under "Known Issues & Future Work" (§13, item KI-1). Until that is resolved, treat any
  value entered into the env-vars form as plaintext-at-rest.
- **Admin JWT tokens** expire after 24 hours. Refresh tokens expire after 7 days.
- **ZIP uploads** must be validated for path traversal (Zip Slip) before extraction. Reject any
  archive entry whose resolved path escapes the target directory.
- **MCP server subprocesses** must never be spawned with shell=True. Always use `subprocess` with
  an explicit list of arguments.
- **No `eval()` or `exec()`** in hub code.
- **The tinyproxy HTTP proxy** is the sole egress channel for MCP server processes. Hub processes
  must NOT set `HTTP_PROXY` on themselves.
- All user-supplied strings used in file paths must be sanitized: alphanumeric + hyphens/underscores
  only for server names and group names.

## 6. API Design

- **REST conventions:** `GET` for reads, `POST` for creates, `PUT` for full updates, `PATCH` for
  partial updates, `DELETE` for removal. No RPC-style verbs in URL paths.
- **Versioning:** All routes are prefixed `/api/v1/`. The `/mcp` endpoint is not versioned (follows
  MCP spec).
- **Response envelopes:**
  - Success: `{ "data": <payload>, "meta": { "timestamp": "..." } }`
  - Error: `{ "error": { "code": <int>, "message": "<str>", "details": <obj|null> } }`
- **Current deployment endpoints:**
  - `POST /api/v1/upload` accepts ZIP packages with `manifest.json`.
  - `POST /api/v1/upload/single-file` creates a package from editor-provided `main.py` code and
    `requirements.txt` content.
- **Runtime parameters:** Per-server `env_vars` are edited through `PATCH /api/v1/servers/{name}`
  and injected into the managed subprocess environment on the next start/restart.
- **Pagination:** List endpoints accept `?page=1&page_size=50` query params. Response includes
  `meta.total`, `meta.page`, `meta.page_size`.
- **Input validation:** All request bodies use Pydantic v2 models with explicit field validators.
  Never trust raw dicts from requests.

## 7. Database

- **ORM:** SQLAlchemy 2.x async (with `AsyncSession`).
- **Migrations:** Alembic. Every schema change requires a migration file. Never use
  `Base.metadata.create_all()` in production code (only in test fixtures).
- **Naming conventions:**
  - Tables: `snake_case`, plural (e.g., `mcp_servers`, `api_keys`).
  - Columns: `snake_case`.
  - Indexes: `ix_<table>_<column>`.
  - Foreign keys: `fk_<table>_<referenced_table>`.
- **Timestamps:** Every table has `created_at` and `updated_at` columns with server-side defaults.

## 8. Docker & Deployment

- **Single `docker-compose.yml`** — no separate dev/prod compose files. Environment-specific
  behavior is controlled entirely via `.env` (which is git-ignored; `.env.example` is committed).
- **Multi-stage Dockerfile:** `builder` stage installs deps; `runtime` stage is minimal.
- **Non-root user:** The hub process runs as `uid=1000 gid=1000` inside the container.
- **No `--privileged`:** Never required. Document this explicitly in README.
- **Health check:** `docker-compose.yml` defines a `healthcheck` for the hub container using
  `curl -f http://localhost:8000/api/health`.
- **Ports:** The container always listens internally on `8000`. The host port is `${HUB_PORT:-8000}`.
  Check `docker compose ps` or `.env` before giving the user a URL.
- **Volumes:**
  - `./data:/app/data` — persistent SQLite DB and log files.
  - `./servers:/app/servers` — deployed MCP server packages.

## 9. Git & Commit Conventions

- **Branching:** `main` is always deployable. Feature work on `feature/<name>`, fixes on `fix/<name>`.
- **Commit messages:** Follow Conventional Commits:
  - `feat(scope): description` — new feature
  - `fix(scope): description` — bug fix
  - `refactor(scope): description` — non-functional change
  - `test(scope): description` — adding/fixing tests
  - `docs(scope): description` — documentation only
  - `chore(scope): description` — tooling, config, deps
- **No commit should break the test suite.** Run tests locally before committing.
- **No committing `.env` files, `data/`, `servers/` contents, or `__pycache__`.**

## 10. Frontend Conventions

- **Component files:** PascalCase filenames matching the component name (`ServerCard.tsx`).
- **One component per file.** Shared sub-components can live in the same file only if they are
  never used elsewhere (and are not exported).
- **API calls:** All backend communication goes through `src/lib/api.ts` — a typed API client.
  Never use `fetch` directly in components or hooks.
- **Theme:** Tailwind v4 dark mode is class-based. Keep `@variant dark (&:where(.dark, .dark *));`
  in `frontend/src/index.css`; the toggle persists `localStorage.theme` and updates the root
  `<html>` class.
- **Error boundaries:** Every page-level component is wrapped in a React error boundary.
- **SSE connections:** Managed via a custom `useSSE` hook. Always clean up the `EventSource`
  on component unmount.
- **Forms:** React Hook Form + Zod for all user-facing forms. No uncontrolled inputs for
  sensitive data.
- **Accessibility:** All interactive elements must have `aria-label` or visible label text.
  No `onClick` on non-interactive elements (use `<button>` not `<div onClick>`).

## 11. MCP Server Package Conventions

MCP servers can be deployed in two ways:
- ZIP upload using the package format documented in `docs/server-manifest.md`.
- Single-file editor creation, which writes `main.py` and `requirements.txt` under
  `servers/<server_name>/` and registers the server with entrypoint module `main`.

Key rules enforced at upload time:

- `manifest.json` must be present at the ZIP root.
- `requirements.txt` must be present.
- The `entrypoint` field in the manifest must point to an existing file in the ZIP.
- Server name must match `/^[a-z0-9][a-z0-9_-]{1,62}[a-z0-9]$/`.
- No absolute paths in the ZIP (Zip Slip prevention).
- Servers communicate via **stdio** using the MCP JSON-RPC protocol.
- Servers must write logs/errors to **stderr** — stdout is reserved for MCP protocol messages.
- Runtime parameters are plain environment variables stored as `env_vars` on `McpServer`; these are
  not a formal MCP protocol feature. They are stored in cleartext in the SQLite DB today (see
  §13 KI-1) — avoid using them for long-lived high-value secrets until that is fixed.

## 12. Code Review Checklist

Before any PR or phase completion, verify:

- [ ] All new functions have type annotations.
- [ ] All new endpoints have at least 2 tests (happy path + error path).
- [ ] No bare `except` clauses without logging.
- [ ] No hardcoded secrets or IPs.
- [ ] `ruff check hub/` passes with zero errors.
- [ ] `mypy hub/` passes with zero errors.
- [ ] `pytest tests/ --cov=hub` shows >= 80% coverage for changed modules.
- [ ] New DB models have a corresponding Alembic migration.
- [ ] API responses follow the envelope format.
- [ ] Error responses include full traceback when originating from an MCP server.
- [ ] Docker build succeeds: `docker compose build`.

## 13. Known Issues & Future Work

Items recorded here are **acknowledged gaps** between the current implementation and the
guidelines in this document, or design decisions deferred for later. Each item has a stable
identifier (`KI-N`) so it can be referenced from code comments, PR descriptions, and other
sections of this file. New items append to the list — never renumber existing ones.

### KI-1 — Per-server `env_vars` are stored in cleartext

**Status:** open
**Affects:** §5 (Security), §6 (API Design), §11 (MCP Server Package Conventions),
`docs/server-manifest.md`
**First flagged:** 2026-05-06 (during Grafana MCP server packaging review)

**Problem.** Per-server runtime parameters edited in the web UI are persisted to
`McpServer.env_vars` as a plain JSON string in the SQLite database (`data/mcp_central.db`).
This includes values declared `"secret": true` in a server's `manifest.json` — for example,
upstream API tokens like `GRAFANA_API_KEY`. Anyone with read access to the SQLite file (host
filesystem, backup snapshots, accidental commits of `data/`) can recover those secrets verbatim.

This contradicts the spirit of §5, which mandates SHA-256 hashing for hub-issued API keys but
is silent on per-server env vars. It also conflicts with the now-corrected wording in
`docs/server-manifest.md` that previously implied secret values lived only in the host `.env`.
The current code path is:

- `hub/api/upload.py` (around line 264) seeds `env_vars` with empty strings keyed by the manifest's
  `env` declarations.
- `hub/api/servers.py` (around lines 124–125) accepts `env_vars` updates from `PATCH
  /api/v1/servers/{name}` and writes them back as JSON.
- `hub/process/manager.py` (around line 427) reads them at start/restart and merges them into the
  subprocess environment via `SandboxConfig`.

**Why we are accepting it for now.** The single-binary, single-host deployment model means the
SQLite file is already as sensitive as the host filesystem itself, and adding encryption without
a clear key-management story (where does the master key live? rotated how? recoverable how?)
risks giving a false sense of security.

**What "resolved" looks like.** Any of the following, in roughly increasing order of effort:

1. **Mark and warn.** Surface a UI banner on every server detail page noting that env vars are
   stored in cleartext; add the same warning to `docs/server-manifest.md`. This is the minimum
   bar and should land first.
2. **Encrypt at rest** for entries whose manifest declares `"secret": true`. Use a Fernet key
   derived from `SECRET_KEY` (already required in `.env` per §5), encrypt only the value half of
   the JSON, and decrypt lazily in `hub/process/manager.py` immediately before subprocess spawn.
   Add a migration that rewrites existing rows. Document the operational consequence: rotating
   `SECRET_KEY` invalidates stored secret env vars.
3. **External secret reference.** Allow values of the form `${env:VAR_NAME}` or `${file:/path}`
   so operators who want zero secrets in the DB can keep them in the host environment or a
   mounted file, and the hub resolves them at spawn time. Combine with #2 for defense in depth.

**Test obligations when this is picked up** (per §3):
- Round-trip unit test: write a secret env var, read it back through the API, assert the API
  response masks the value (e.g. returns `"***"` or omits it) for `secret: true` entries.
- DB-level test: assert the raw SQLite cell contents for a `secret: true` entry are not equal to
  the plaintext that was written.
- Process manager integration test: assert the subprocess receives the correct plaintext value in
  its environment despite at-rest encryption.
- Migration round-trip: existing cleartext rows must be readable and re-encrypted on first access.

**Do not silently change behavior.** Any fix must keep the UI form working unchanged from the
operator's perspective. The fix is about how the hub stores the value, not how it is entered.
